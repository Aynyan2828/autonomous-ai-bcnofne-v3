# LLM 実行ポリシー (LLM Runtime Policy)

本ドキュメントは、BCNOFNe システムにおける LLM 呼び出しの標準挙動と異常系の処理方針を定義します。

## 1. タイムアウト方針
- **標準 (Standard)**: 60秒。通常の会話や分類に使用。
- **重い処理 (Heavy)**: 120秒以上。航海日誌の要約、長期記憶の整理、コード生成に使用。
- **急速 (Rapid)**: 30秒以内。埋め込み生成、単純なキーワード抽出に使用。

## 2. リトライとバックオフ
- **Ollama (Local)**:
    - 最大リトライ回数: 3回
    - 指数バックオフ: 2.0^attempt (1s, 2s, 4s...)
    - 理由: Raspberry Pi のリソース競合による一時的なエラーを許容するため。
- **OpenAI (Cloud)**:
    - 最大リトライ回数: 2回
    - 理由: ネットワークエラーやレート制限への対応。

## 3. フォールバック (Fallback) 階層
1.  **第一優先: Ollama (qwen2.5:7b)**
    - 原則として全ての推論をローカルで行う。
2.  **第二優先: OpenAI (gpt-4o-mini)**
    - Ollama が3回リトライしても失敗した場合、または Ollama への接続自体が拒否された場合に発動。
    - `FALLBACK_ENABLED=true` の場合のみ有効。
3.  **最終手段: エラー応答**
    - クラウド側も失敗した場合、または `FALLBACK_ENABLED=false` の場合は、ユーザーに「あゆにゃん、ちょっと考え込みすぎて思考がまとまらんかったばい...」等のエラー通知を送信する。

## 4. JSON 修復 (JSON Repair)
- 7B モデルは JSON の閉じタグを忘れたり、Markdown の囲みを付けて返したりすることがある。
- `LLMExecutor` 内部の `json_repair.py` および `repair` タスクプロンプトを用いて、2段階のパースを試みる。
- 修復不能な場合は `ValueError` を送出し、リトライまたはフォールバックに回す。
